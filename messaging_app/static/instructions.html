What is Tari Messenger? A proof-of-concept blockchain messaging app

Tari Messenger is a WhatsApp-style chat application where every message is a real blockchain transaction. There is no central server storing your messages — they live on the Tari Ootle Layer 2 network, an experimental testnet built on Rust and WASM smart contracts.

You can create multiple wallets, send direct messages (DMs) to any public key, create group chat rooms, back up and restore keys, and (experimentally) enable end-to-end encryption for DMs using Ristretto ECDH + AES-256-GCM.

💡 No real money involved. This runs on the Esmeralda testnet. All tokens (tTARI) are free test tokens from a faucet — zero real-world value. Great for learning how blockchain works in practice.

Wallet Management Creating, importing, backing up, and switching wallets

A "wallet" is a Ristretto elliptic-curve key pair — an account secret key and a view key, both 32 bytes (64 hex chars). The public key derived from the account secret key is your on-chain identity.

✨ Generate New Wallet

Creates a random key pair and immediately requests 10 tTARI from the Esmeralda faucet. Fastest way to start. The keys are saved to messaging-state.json.

🔑 Import by Keys (hex)

Paste your existing 64-char hex account secret key and view secret key. Use this to import a wallet you backed up previously. Your wallet must already have tTARI — use 💧 Faucet in the HUD if needed.

📝 Import by Passphrase

Derives keys from a passphrase using SHA-256. The same passphrase always produces the same wallet. Not compatible with walletd/daemon seed words — uses a different derivation scheme. Good for deterministic test wallets.

🌱 Import by Seed Words (BIP-39)

Paste the 24 BIP-39 seed words from the 🔑 Backup screen. The account and view keys each have separate seed word sets. This is the recommended backup/restore format.

🔑 Backup Your Keys

Click 🔑 Backup in the wallet info panel to see your public key, account address, raw hex private keys, and BIP-39 seed words. Store these somewhere safe — losing them means losing your wallet. Anyone with your private keys has full control of your funds.

Switching Wallets

Click the wallet name top-left to open the dropdown. Click any wallet to switch. Each wallet has its own inbox, DM history, and balance. Switching wallets reloads the conversation list for that wallet.

💧 Faucet

Click 💧 Faucet in the wallet HUD to request more free tTARI from the testnet faucet. Useful if your balance runs low after many messages. Takes ~30 seconds to confirm.

Messaging Direct messages and group rooms

Direct Messages (DMs)

Click ✉ New DM, paste the recipient's 64-char Ristretto public key, and optionally give them a nickname. They can find their public key in the wallet info panel (PK field). Once open, you can type messages and press Enter. Messages are stored on-chain with the sender's public key (set by the blockchain — cannot be faked) and the recipient's public key (you provide).

Group Rooms

Click 👥 New Group, enter a unique Room ID (e.g. general), and a display name. The room is created on-chain. Share the Room ID with others so they can join via 🔗 Join Room. Anyone who knows the Room ID can post — no invite needed. All room messages are always public (no encryption in group rooms).

Why do messages show up instantly?

The UI adds your message to the local cache immediately for a snappy feel. The blockchain transaction runs in the background (~10–60 seconds). When it confirms, the Debug Panel shows a Committed entry. Messages from the chain are synced automatically every ~10 seconds via the Esmeralda indexer.

Contact Names

Click ✏️ Set Name in the chat header to give a contact a nickname. Names are stored locally in your state file — they're not on the blockchain. Use them to remember who's who.

Sync Button

The app polls the chain every 10 seconds automatically. If you want to force a sync, open the Debug Panel → Network tab → hit Sync, or call POST /api/sync directly.

End-to-End Encryption (Experimental) Ristretto ECDH + HKDF-SHA256 + AES-256-GCM

Tari Messenger includes an experimental E2EE encryption feature for direct messages. When enabled, message content is encrypted on the server before being stored on-chain. Even though the ciphertext is publicly visible on a blockchain, only the sender and recipient can decrypt it.

⚠ This is an experimental feature. It works correctly for DMs between wallets sharing the same app instance (same messaging-state.json). See the limitations section below.

🔒 Encryption Protocol

1
ECDH Key Agreement (Ristretto255)
shared_point = recipient_pubkey × sender_secretkey
Both the sender (using their sk × recipient pk) and the recipient (using their sk × sender pk) compute the same shared point. This is standard Diffie-Hellman on an elliptic curve — the same cryptography used by Signal, WhatsApp, and TLS.
2
Key Derivation (HKDF-SHA256)
key = HKDF-SHA256(ikm=shared_point, info="tari-dm-v1:" + sorted(pk_a, pk_b))
The 32-byte shared point is passed through HKDF to produce a 256-bit symmetric key. Public keys are sorted before being used as the HKDF info string — this ensures the sender and recipient always derive the same key regardless of who initiates.
3
Authenticated Encryption (AES-256-GCM)
ciphertext = AES-256-GCM.encrypt(key, nonce=random_12_bytes, plaintext)
Each message gets a fresh random 12-byte nonce. AES-GCM is an authenticated encryption mode — it detects tampering. If any byte of the ciphertext is changed, decryption fails and the message shows as undecryptable.
4
Wire Format
ENC1:<hex(nonce[12] || ciphertext)>
This string is stored on-chain. Anyone can see it's encrypted, but cannot read the content. Messages without the ENC1: prefix are always treated as plaintext.
// On-chain stored message (example) "ENC1:a3f2b89c1d4e...f7a2b3c4d5e6f7a8b9c0d1e2f3a4b5c6d7e8f9a0b1c2d3e4f5" // └─ 12-byte nonce hex ─┘└──────── AES-GCM ciphertext hex ─────────────────┘ // Decryption (using recipient's wallet key) shared_point = sender_pubkey × recipient_secretkey key = HKDF(shared_point, info="tari-dm-v1:alice_pk:bob_pk") plaintext = AES-256-GCM.decrypt(key, nonce, ciphertext)

Enable Encryption

In the wallet info panel (bottom-left), toggle 🔒 E2EE Encryption (experimental). All new DMs you send will be encrypted. Received encrypted DMs will automatically be decrypted if you have the relevant wallet loaded. Successfully decrypted messages show a 🔒 lock icon.

Limitations

⚠ Group chats are always plaintext

ECDH requires exactly two parties — a sender and a recipient. Group rooms can have any number of members, so there is no shared secret that works for all of them. Group room messages are never encrypted, even with E2EE enabled. A warning bar appears when you enter a group room with encryption on.

⚠ Both wallets must share the same app instance

Decryption requires the wallet's private key. If Wallet A sends an encrypted DM to Wallet B, and Wallet B is loaded in the same app instance (same messaging-state.json), the app will automatically decrypt it. If Wallet B is on a different machine, it needs to import the key and have a compatible Tari Messenger running to decrypt.

⚠ Metadata is not hidden

The blockchain always records the sender's public key, recipient's public key, and timestamp. Only the content is encrypted. A blockchain observer can still see who is talking to whom and when.

⚠ No forward secrecy

This implementation uses static ECDH — the same conversation key is derived from the same key pair. If your private key is compromised, all past encrypted DMs can be decrypted. True forward secrecy (like Signal's double ratchet) would require per-message ephemeral keys, which is a future enhancement.

⚠ Old messages are not re-encrypted

Toggling encryption off does not change already-stored encrypted messages. They remain as ENC1:... on-chain and in local state. Toggling encryption back on resumes encrypting new messages only.

How a message travels From keypress to blockchain commitment

When you press Enter or click ➤, here is every step that happens, in order.

1
Browser → Web Server POST /api/dm/send

The chat UI sends a JSON request to the local Axum web server with the sender's public key, recipient's public key, and the plaintext message content.

↓
2
Content Encryption (if enabled) 🔒 E2EE

If encryption is enabled, the server performs ECDH using the sender's stored private key and the recipient's public key to derive a shared AES-256-GCM key. A random 12-byte nonce is generated, the content is encrypted, and the result is formatted as ENC1:<hex>.

If encryption is off, the content is stored and sent as-is (plaintext).
↓
3
Local Cache + UI Update Instant feedback

The message is immediately added to the server's in-memory AppState.dms list and saved to messaging-state.json. The browser's next API poll renders it in the chat. This makes the UI feel instant while the blockchain confirms in the background.

↓
4
Build Transaction TransactionBuilder

A background thread builds an unsigned transaction with two instructions:

pay_fee_from_component(sender_account_address, 2_000 µTARI) call_method(component_address, "send_dm", [recipient_pubkey_hex, content])

Input references to the sender's fee vault and the messaging component are resolved from the indexer.

↓
5
Sign the Transaction Sender's private key

The unsigned transaction is signed with the sender's Ristretto private key. This proves to the network that the sender authorised this exact transaction. The signature covers every byte — it cannot be tampered with after signing.

🔒 The private key never leaves the local machine. Only the signed transaction is sent to the network.
↓
6
Submit to Indexer Esmeralda testnet

The signed transaction is broadcast to ootle-indexer-a.tari.com, which forwards it to Tari Layer 2 validators. The app watches the transaction — polling until it is committed or rejected (typically 10–60 seconds on testnet).

↓
7
Validators Execute the WASM Smart Contract On-chain

Validator nodes run the messaging WASM template. Inside the contract:

pub fn send_dm(&mut self, to: String, content: String) { // "from" is authenticated by the blockchain — cannot be faked let from = CallerContext::transaction_signer_public_key().to_string(); self.msg_from.push(from); self.msg_to.push(to); self.msg_content.push(content); // ENC1:... or plaintext self.next_id += 1; }

The from field is set by the network — not the caller. No one can spoof the sender.

↓
8
State Committed + Decryption on Read 🔒 E2EE

The message (encrypted or not) is permanently stored in the component's on-chain state. When a recipient queries GET /api/dm/messages, the server attempts to decrypt using any loaded wallet key that matches the sender or recipient. Successfully decrypted messages return with "encrypted": true in the API response — the UI shows a 🔒 badge.

🔑 Your On-Chain Identity

There is no username or email. Your identity is your Ristretto public key — a 32-byte elliptic curve point shown as a 64-character hex string:

3a8f2c1d...b9e4f2a1 (64 hex chars = your address on Tari)

This key is derived from your private key using Ristretto255 scalar multiplication. Every on-chain message uses this key as the from or to address. The human-readable display name ("Alice") only exists in the local app state — the blockchain only knows public keys.

Encryption adds another layer: the same key pair used for on-chain identity is also used for ECDH key agreement. This means no separate encryption keys are needed — your wallet IS your encryption identity.

Token Costs & Fees What each action costs in tTARI

Every on-chain action costs a fee in micro-TARI (µTARI). 1 tTARI = 1,000,000 µTARI. Each new wallet starts with 10 tTARI (10,000,000 µTARI) from the faucet.

Action Fee (µTARI) Fee (tTARI) Notes
Fund from faucet 500 0.0005 You receive 10 tTARI (10,000,000 µTARI)
Publish template ~250,000 0.25 One-time; proportional to WASM size
Deploy component 2,000 0.002 One-time per messaging component
Send DM 2,000 0.002 Per message, paid by the sender
Create room 2,000 0.002 One-time per room
Post to room 2,000 0.002 Per post, paid by the poster

📊 Budget example: With 10 tTARI, after the initial setup (~252,000 µTARI), you have ~9,748,000 µTARI left — enough for about 4,874 messages. You can always refill with 💧 Faucet.

The Debug Panel (🔍 icon, top-right) shows every transaction with its type, status, fee paid, and the transaction ID.

Blockchain Concepts Explained Plain-language guide for newcomers

📜

Template (Smart Contract)

A Rust program compiled to WebAssembly (WASM) that runs on the blockchain. Defines what the messaging service can do: store messages, check identities, return conversations.

🏠

Component (Contract Instance)

A deployed, live copy of a template with its own on-chain state (the actual messages). The template is the code; the component is the running instance.

🔑

Wallet / Account

A cryptographic key pair. The account secret key signs transactions. The public key is your on-chain identity — it can't be faked. Losing the key = losing the wallet.

💸

tTARI Tokens

The native testnet currency, used to pay fees. Free from the faucet. No real-world value. This app uses Esmeralda testnet tTARI only.

📦

Transaction

Any action that changes on-chain state. Each is signed with your private key, broadcast to the network, verified by validators, and permanently recorded on-chain.

🔍

Indexer

A service that indexes the blockchain and makes it queryable. This app connects to ootle-indexer-a.tari.com to submit transactions and read component state.

🔒

ECDH

Elliptic Curve Diffie-Hellman — a protocol for two parties to derive the same shared secret using only their public keys and their own private key. The basis of the E2EE feature here.

🌱

BIP-39 Seed Words

A 24-word mnemonic phrase encoding your private key as human-readable words. Easier to write down and restore than raw hex. Each key (account + view) has its own seed word list.

Templates & Components How the smart contract actually works

What is a Template?

A template is the smart contract — the code that runs on the Tari Layer 2 blockchain. It's written in Rust, compiled to WebAssembly (.wasm), and uploaded to the network once. After that, anyone can use it.

Think of a template like a class definition in programming. It defines the structure and behaviour — what data it holds, what methods it exposes, what rules it enforces — but it isn't a live running thing by itself.

For this app the template lives in messaging_template/src/lib.rs. It exposes methods like send_dm, create_room, send_room_post, and query methods to read messages back. The from field on every message is set by the network itself using the signer's verified public key — it cannot be faked or spoofed by passing a different value in.

// Simplified template structure #[template] mod messaging_service { pub struct MessagingService { dm_from: Vec<String>, // sender public key dm_to: Vec<String>, // recipient public key dm_content: Vec<String>, // message content (or ENC1:... if encrypted) dm_timestamp: Vec<u64>, // unix seconds rooms: Vec<String>, // room IDs room_posts: Vec<String>, // room message bodies // ... } impl MessagingService { pub fn send_dm(&mut self, to: String, content: String) { // The network fills in `from` — the caller cannot lie about their identity let from = CallerContext::transaction_signer_public_key().to_hex(); self.dm_from.push(from); self.dm_to.push(to); self.dm_content.push(content); self.dm_timestamp.push(now()); } } }

What is a Component?

A component is a live, deployed instance of a template — like creating an object from a class. It has its own unique address on-chain and its own state (the actual stored messages, rooms, and posts).

When you first set up this app, it deploys a messaging component from the template. That component gets a permanent address like component_0e69766efc95e.... Every DM and room post you send is a transaction that calls a method on that specific component and updates its state permanently.

The relationship is:

  • Template = the blueprint (uploaded once, shared by all)
  • Component = your live instance (unique address, your data)

How the App Uses the Template

Here is the exact sequence when you send a DM:

  1. The web server builds a transaction: call_method(component_address, "send_dm", [recipient_pk, content])
  2. The transaction is signed with the sender's private key (held in the server, never the browser)
  3. The signed transaction is submitted to the Esmeralda indexer
  4. Validator nodes on the network receive it, run the WASM template code, verify the signature, and if valid, commit the state change to the component
  5. The component's on-chain state now contains the new message — permanently and immutably
  6. The background sync task reads it back via the indexer REST API and shows it in the UI

The component address is stored in the app's state file. If you lose it, the messages still exist on-chain — you just need the address to find them again.

Running Your Own Instance

Because templates are shared code on the network, anyone can deploy their own component from the same template and have a completely independent messaging instance. Your component's state is separate from everyone else's — no one can read or write to it unless they know your component address and have a valid wallet.

To run your own instance:

  1. Build the WASM — cd messaging_template && cargo build --target wasm32-unknown-unknown --release
  2. Publish the template — go to ⚙ Settings → Configure Template → upload the .wasm file and click Publish. This costs ~250,000 µTARI in fees and takes ~60 seconds to confirm on-chain. You get back a template address.
  3. Deploy a component — the app automatically calls call_function(template_address, "new", []) to create a fresh component instance. You get a component address.
  4. Share the component address — anyone who wants to join your messaging network can go to ⚙ Settings → Join Existing and paste your component address. They'll see the same messages and can send into the same component.

Alternatively, you can skip publishing and join a pre-existing component that someone else already deployed — you'll share the same message space with them. The component address is the only thing you need to join.

Why Deploy Your Own vs. Joining an Existing One?

Your Own Component Join Existing
Full isolation — your messages, your network Shared space with whoever deployed it
Costs ~250,000 µTARI to publish template Free — just paste a component address
You control who gets the address Anyone with the address can post
~60 seconds to deploy on Esmeralda testnet Instant — no transaction needed

Modifying the Template

Because the template source is in messaging_template/src/lib.rs, you can modify it — add new message fields, change access rules, add new room features — recompile, publish a new template, and deploy a fresh component from your modified version. The old component keeps running with the old code; you'd start fresh with a new component address.

This is the core of how Tari Ootle works: anyone can fork the code, publish their own template, and run a completely independent instance with custom behaviour — no permission needed from anyone.

Code Architecture For the curious developer

messaging_template/ — The Smart Contract

Written in Rust, compiled to .wasm with cargo build --target wasm32-unknown-unknown --release. Uses Tari's tari_template_lib crate. Stores messages as parallel Vec<String> fields (from, to, content, timestamp) and exposes methods: send_dm, send_room_post, create_room, and query methods. The from field is always set by CallerContext::transaction_signer_public_key() — the network's verified identity, not a parameter.

messaging_app/ — The Web Server & UI

An Axum HTTP server (Rust, async, Tokio) serving the WhatsApp-style frontend and a REST API on port 3000. Uses ootle-rs to sign and submit transactions. Because IndexerProvider is !Send, all blockchain operations run in dedicated OS threads with single-threaded Tokio runtimes (spawn_blocking + new_current_thread).

Encryption Layer — Server-side

AES-256-GCM encryption/decryption happens entirely in the Rust server (messaging_app/src/main.rs). The private keys are never sent to the browser. The encryption helpers use: tari_crypto (Ristretto ECDH), hkdf (key derivation), aes-gcm (symmetric encryption), and rand (nonce generation). All are pure Rust crates with no native dependencies.

State Persistence

All app state is saved to a JSON file (messaging-state-<name>.json by default). Contains all wallet keys, template/component addresses, all local DMs, room messages, contacts, encryption setting, and the transaction history. On restart, the app loads this file and resumes exactly where it left off — no re-setup needed.

Chain Sync

A background task polls the Esmeralda indexer REST API every 10 seconds via GET https://ootle-indexer-a.tari.com/substates/{component_id}. The component's on-chain state is CBOR-encoded JSON. The server parses it and imports any messages not already in the local cache (using a set of synced_event_ids to avoid duplicates).

Troubleshooting Common issues and fixes

Setup is stuck / "Error" status

The Esmeralda testnet can be slow or occasionally unavailable. Wait 60–90 seconds and refresh. If it stays stuck, check the Debug Panel for error details. As a last resort, delete the state file and restart the app to start fresh.

"WASM not found" when publishing

Run cd messaging_template && cargo build --target wasm32-unknown-unknown --release first. The launch.bat script does this automatically.

Transaction rejected / insufficient fees

Your wallet may be out of tTARI. Click ↺ Balance in the wallet HUD to check, then 💧 Faucet to refill. If fees still fail, the testnet may be congested — try again in 30 seconds.

Messages appear locally but "Encrypted message — key not available"

The recipient's wallet is not loaded in this app instance. Import the recipient's wallet (via keys or seed words) so the app can decrypt the messages. Both the sender and recipient wallet need to be in the same messaging-state.json for decryption to work automatically.

Balance shows "Unknown" or stuck

Click ↺ Balance manually. The balance is fetched from the indexer REST API, which can be slow. If it's still failing, check GET /api/wallet/balance?wallet_pk=<pk> in the browser for the raw error.

Messages not syncing from chain

The chain sync runs every 10 seconds. Force a sync via Debug Panel → Network tab → Sync button, or call POST /api/sync. If the component address is wrong, check the Settings modal and re-configure/join with the correct component address.

Sharing the app with others

Others need to run their own instance of the app. Share your component address (from ⚙ Settings → Join Existing) so they can join the same messaging component. They also need to know your public key (from the wallet HUD) to DM you.
