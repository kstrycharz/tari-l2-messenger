Every message in this app is a real blockchain transaction on the Tari Layer 2 network. Your wallet's private key signs every message — proving it came from you.

Quick Start
1. Add a wallet — click your name top-left → Add Wallet → Generate
2. Configure template — click the yellow banner → Auto-Publish WASM
3. Wait ~60s for the component to deploy
4. Start chatting — New DM or New Group
🧪 Demo Mode

Creates two wallets named Ootle and Minotari, funds them from the testnet faucet, and opens a DM conversation between them so you can test two-way messaging.

What is a Direct Message?
A DM is stored on-chain in the messaging component with the sender's public key (set by the blockchain — can't be faked) and the recipient's public key (you provide). By default, DMs are plaintext. Enable 🔒 E2EE Encryption in the wallet info panel to encrypt them end-to-end using Ristretto ECDH + AES-256-GCM.
How to send a DM
1. Click ✉ New DM in the sidebar
2. Paste the recipient's public key hex (64 chars) — they can find it in the wallet selector dropdown
3. Optionally give them a nickname
4. Type and hit Enter — message appears instantly (blockchain confirms in ~30-60s in background)
What happens on-chain
component.send_dm(to_pubkey, "hello")
from = CallerContext::signer_public_key() // can't spoof
fee: 2,000 µTARI from your account
What is a Group Room?
A room is created on-chain with a unique room_id (like "general"). Anyone who knows the room_id can post to it — no invite needed. All messages are public. The creator pays 2,000 µTARI to create it; posters pay 2,000 µTARI per post.
Create a room
1. Click 👥 New Group
2. Enter a unique Room ID (no spaces, e.g. general)
3. Enter a display name (e.g. General Chat)
4. Hit Create — transaction runs in background (~30-60s)
Join an existing room
1. Click 🔗 Join Room
2. Enter the room_id someone shared with you
3. No transaction needed — joining is just local bookkeeping
4. You can post immediately once joined
Why does sending take ~30-60 seconds?
The message is submitted as a signed transaction to the Esmeralda testnet indexer. Validators on the network confirm it, execute the WASM smart contract, and commit the new state. This takes 1-3 block times.

The UI shows the message instantly (from local cache) while the blockchain confirms in the background. Check the 🔍 Debug panel to see when it confirms.
Your identity on-chain
Your identity is your Ristretto public key — a 32-byte elliptic curve point, shown as 64 hex characters. There are no usernames or accounts — just cryptographic keys. The blockchain itself verifies that the sender signed the transaction.
Are messages private?
Partially. With E2EE enabled (🔒 toggle in the wallet panel), DMs are encrypted with AES-256-GCM before being stored on-chain. Group chat messages are always plaintext. Without E2EE, all messages are stored as plain text and readable by anyone with the component address.
🔒 Experimental E2EE Encryption
When enabled, all new DMs are encrypted before being stored on-chain. Even though the ciphertext is on a public blockchain, only the sender and recipient can decrypt it.
How it works
1. ECDH Key Agreement
  shared_point = recipient_pubkey × sender_secretkey
  (Ristretto255 elliptic curve — same math as the wallet keys)
2. Key Derivation
  HKDF-SHA256 over the shared point → 256-bit symmetric key
  Sorted public keys used as info — ensures sender & recipient derive the same key
3. Encryption
  AES-256-GCM with a random 12-byte nonce per message
  Wire format: ENC1:<hex(nonce||ciphertext)>
Limitations
⚠ Group chats are always plaintext — ECDH requires exactly two parties, so it can't be used for groups.
⚠ Both parties must be on the same app instance (or share state files) to decrypt each other's messages.
⚠ Metadata is not hidden — the sender's public key and recipient's public key are always visible on-chain.
⚠ Old messages stay encrypted — switching encryption off doesn't re-encrypt old messages.
Enable It
In the wallet info panel (bottom-left), toggle 🔒 E2EE Encryption (experimental). New DMs will show a lock icon 🔒 when successfully encrypted and decrypted.

DMs can be end-to-end encrypted so only you and your recipient can read the content — even though ciphertext is stored on a public blockchain.

🤝
Step 1 — ECDH Key Agreement
Your wallet's private key and your contact's public key are multiplied together on the Ristretto255 elliptic curve:shared_point = recipient_pubkey × sender_secretkeyBoth sides compute the same shared point — without ever exchanging private keys.
🔑
Step 2 — Key Derivation
The shared point is fed into HKDF-SHA256 to derive a 256-bit symmetric key:key = HKDF-SHA256(shared_point, info=sorted_pubkeys)Sorted public keys in the info field ensure sender and recipient always derive the same key regardless of who sends first.
🔒
Step 3 — Encrypt
AES-256-GCM with a fresh random 12-byte nonce per message:wire: ENC1:<hex(nonce || ciphertext)>The ENC1 prefix tells the receiver it needs to decrypt rather than display raw text.
⚠
Limitations: Even with E2EE, metadata (sender/recipient keys, timestamps) is always visible. Group chats can never be encrypted. Both parties must be on this app to decrypt. Toggle E2EE in the wallet info panel (bottom-left of sidebar).

Want more privacy? Deploy your own messaging contract. Only people you give the address to can read your messages.

🏗
How it works: A "messaging contract" is a Rust WASM smart contract deployed to the Tari Layer 2 network. It stores messages on-chain. You deploy it, you get its address, and only users who have that address can post to or read from it.
🚀
Simple path — Auto-Publish: Click ⚙ Settings in the sidebar, then Auto-Publish WASM. The app compiles and deploys the contract automatically. Takes ~60 seconds and costs a small testnet fee.
🔗
Share with contacts: After deployment, open ⚙ Settings to see your component address. Share it with anyone you want to chat with — they paste it into "Join Existing" on their end.
⚙
Advanced path — custom template: The Rust source is in messaging_template/src/lib.rs. You can modify the contract logic, add new methods, or tweak access rules — then publish your customized version via the Tari Wallet Web UI at http://127.0.0.1:5100.
